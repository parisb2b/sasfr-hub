2. CAUSE RACINE CONFIRMÉE
──────────────────────────
Le fichier firestore.rules du projet LISTES ne couvrait que 3 ensembles :
  • users/{userId}
  • listes_lists/{listId}
  • listes_lists/{listId}/tasks/{taskId}

Il MANQUAIT toutes les règles pour :
  • comptes_client_sessions → TOTAL (propriétaire, collaborateurs, admin)
  • comptes_client_trash     → TOTAL (corbeille)
  • chantiers + 4 sous-collections → DM (membres, tâches, notifications, liste_jour)
  • membres (racine)         → DM (annuaire inscription)
  • admins                   → DM (admin global)

Résultat : l'admin parisb2b@gmail.com voyait "Missing or insufficient permissions"
sur total.sasfr.com car les règles n'autorisaient plus l'accès à
comptes_client_sessions.

3. COLLECTIONS COUVERTES ET LOGIQUE D'ACCÈS
─────────────────────────────────────────────

👥 users (PARTAGÉE — TOTAL + LISTES + DM)
   • read  : tout utilisateur authentifié
   • create: propriétaire (uid == userId)
   • update: admin OU propriétaire
   • delete: admin uniquement

📊 comptes_client_sessions (TOTAL)
   • read  : admin OU ownerId == uid OU email in collaborators
             OU email == clientEmail
   • create: admin OU request.resource.data.ownerId == uid
   • update/delete: admin OU resource.data.ownerId == uid

📊 comptes_client_trash (TOTAL)
   • read  : admin uniquement
   • create: tout utilisateur authentifié
   • update/delete: admin OU resource.data.ownerId == uid

📋 listes_lists (LISTES)
   • create: auth + request.resource.data.createdBy == email
   • read  : admin OU createdBy == email OU email in assignees
   • update: admin OU createdBy == email
   • delete: admin uniquement

📋 listes_lists/{listId}/tasks (LISTES)
   • create/read/update: canAccessList(listId) → admin OU créateur OU assigné
   • delete: admin OU créateur de la liste parente

🏗️ chantiers (DM)
   • read  : estMembreChantier(chantierId)
   • write : admin uniquement

🏗️ chantiers/{id}/membres (DM)
   • read  : estMembreChantier(chantierId)
   • write : admin uniquement

🏗️ chantiers/{id}/taches (DM)
   • read/create/update/delete : estMembreChantier(chantierId)

🏗️ chantiers/{id}/notifications (DM)
   • read  : admin OU destinataireUid == uid
   • create: estMembreChantier(chantierId)
   • update: admin OU destinataireUid == uid
   • delete: admin uniquement

🏗️ chantiers/{id}/liste_jour (DM)
   • read/create/update: estMembreChantier(chantierId)
   • delete: admin uniquement

🏗️ membres — racine (DM, annuaire inscription)
   • read  : admin OU uid == membreId
   • create: uid == membreId
   • update: admin OU uid == membreId
   • delete: admin uniquement

🏗️ admins (DM)
   • read  : uid == uid (lecture de son propre doc)
   • write : interdit (Admin SDK / console Firebase uniquement)

🔐 FONCTION isAdmin() — 3 méthodes combinées :
   1. Email hardcodé : parisb2b@gmail.com (TOTAL + DM)
   2. Champ admin dans users/{uid}.admin == true (LISTES)
   3. Document dans admins/{uid} (DM)

5. INDEX
────────
✅ Aucun nouvel index requis. La requête collaborators utilise
   array-contains sur un seul champ → pas d'index composite
   nécessaire. L'erreur était permission-denied, pas
   failed-precondition.

   Index existants (firestore.indexes.json CHANTIER-TRACKER) :
   • taches: statut ASC + dateButoir ASC
   • taches: assigneA ARRAY_CONTAINS + dateButoir ASC
   • notifications: destinataireUid ASC + createdAt DESC
   • notifications: destinataireUid ASC + lue ASC
   • membres: actif ASC + createdAt ASC
   • taches: archived ASC + archivedAt DESC
